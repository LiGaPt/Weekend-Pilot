# Spec: 078 Public Demo Progress Contract v0

## 1. Goal

Add the first public-safe progress contract to the Web demo API by mapping existing workflow node history, ordered tool events, and demo continuation history into a small fixed set of user-visible stages.

After this task, every successful `/demo/runs*` response must include one additive `progress` object inside `DemoRunSummary`. That object must expose a stable stage enum, a stable customer-facing Chinese label, and an ordered public stage history that the chat-first customer surface can trust. This task closes the backend contract gap exposed by Task `077`, where the UI can already show a conversation-style system-progress row but the backend still exposes only coarse terminal states.

This task does **not** make `POST /demo/runs` asynchronous, does **not** stream progress live during the initial request, and does **not** add a polling endpoint. It is a contract-first slice only.

## 3. Requirements

- Add one additive `progress` object to `DemoRunSummary`.
- The additive `progress` object must be present on successful responses from:
  - `POST /demo/runs`
  - `GET /demo/runs/{run_id}`
  - `POST /demo/runs/{run_id}/clarify`
  - `POST /demo/runs/{run_id}/replan`
  - `POST /demo/runs/{run_id}/confirm`
  - `POST /demo/runs/{run_id}/decline`
- Define one public progress stage enum with exactly these values:
  - `understanding_request`
  - `planning_queries`
  - `searching_activities`
  - `searching_dining`
  - `checking_availability`
  - `building_itinerary`
  - `checking_route_time`
  - `reviewing_plan`
  - `ready_for_confirmation`
  - `executing_confirmed_actions`
- `progress` must include:
  - `schema_version`
  - `current_stage`
  - `current_label`
  - `stage_history`
- `current_label` must be generated by the backend from the public stage enum and must use stable customer-facing Chinese copy.
- `stage_history` must be ordered, deduplicated, and limited to the public progress stage enum values only.
- Build `progress` from existing persisted backend evidence only:
  - `AgentRun.status`
  - `AgentRun.metadata_json["demo"]["initial_node_history"]`
  - `AgentRun.metadata_json["demo"]["continuation_history"]`
  - ordered `ToolEvent` rows for the run
  - existing execution / feedback state already visible through the selected plan summary
- Do not add a new persisted progress table, a new Alembic migration, or a new required metadata blob in this task.
- Do not make `GET /demo/runs/{run_id}` depend on Redis-only ephemeral progress keys in this task. The contract must remain reconstructable from persisted run data after the run completes.
- Map workflow node history to public stages with these exact rules:
  - `initialize`, `parse_intent`, `load_memory` -> `understanding_request`
  - `generate_queries` -> `planning_queries`
  - `pre_flight_check_availability` -> `checking_availability`
  - `logical_planner_agent` -> `building_itinerary`
  - `route_and_time_engine` -> `checking_route_time`
  - `semantic_validator`, `apply_recovery`, `final_review`, `present_to_user` -> `reviewing_plan`
  - `wait_confirmation` -> `ready_for_confirmation`
- Detect `searching_activities` and `searching_dining` from ordered `search_poi` tool events:
  - for Mock World, use `tool_events.request_json.payload.category`
  - for AMap preview runs, use `tool_events.request_json.payload.canonical_category`
- Do not expose separate public stages for `check_weather`, `get_poi_detail`, `check_opening_hours`, `check_queue`, `check_table_availability`, `check_ticket_availability`, or `check_route`.
- Confirmation and execution mapping must use demo continuation history:
  - if continuation history includes `confirm_plan`, `saga_execution_engine`, or `generate_summary_message`, `current_stage` must resolve to `executing_confirmed_actions`
- Decline mapping must not advance the public progress stage into execution:
  - if the run is declined without execution, `current_stage` must remain `ready_for_confirmation`
- Clarification mapping must not invent a new public stage enum value:
  - when `status == "awaiting_clarification"`, keep using the existing top-level `clarification` object for the follow-up prompt
  - `current_stage` must reflect the last reached planning stage instead of introducing a new progress stage
- Legacy and malformed-run fallback must be safe:
  - if `initial_node_history` or `continuation_history` is missing or malformed, build the best safe snapshot from `run.status`, execution state, and usable tool events instead of failing the request
  - missing or malformed tool events must be ignored rather than crashing the response
- The additive `progress` contract must not change:
  - request body shapes
  - status-code behavior
  - clarification behavior
  - replan behavior
  - selected-plan behavior
  - confirmation rules
  - decline rules
  - AMAP confirmation gating
- Update `README.md` and `docs/WEB_DEMO_README.md` to document the additive `progress` object and to state explicitly that live mid-request progress transport is still a later follow-up task.
- Do not add frontend UI changes, polling, SSE, WebSockets, background jobs, or a new public API route in this task.

## 4. Non-goals

- Do not implement asynchronous `POST /demo/runs`.
- Do not add a progress polling route, SSE route, WebSocket route, or background job runner.
- Do not change the customer UI to consume the new field yet.
- Do not change internal observability schemas, pages, or routes.
- Do not change benchmark reports, replay contracts, workflow routing, or Tool Gateway behavior.
- Do not add database tables, migrations, or package dependencies.
- Do not expose raw workflow node names, raw tool names, raw provider names, raw event IDs, trace IDs, session IDs, prompts, or payload bodies in the public contract.
- Do not commit `.env`, API keys, tokens, secrets, generated runtime artifacts, or unrelated local files.

## 5. Interfaces and Contracts

### Inputs

This task depends on existing persisted demo and workflow evidence only:

- `AgentRun.status`
- `AgentRun.metadata_json["demo"]["initial_node_history"]`
- `AgentRun.metadata_json["demo"]["continuation_history"]`
- selected plan state already used by `DemoWorkflowService.build_summary(...)`
- ordered `ToolEventRepository.list_for_run(run_id)` results
- existing public `clarification`, `plan_version`, plan preview, execution, and feedback summaries

No new user request field, no new frontend request field, and no new environment variable is required.

### Outputs

Additive public API output:

- `DemoRunSummary.progress`

No new route is added.

### Schemas

`DemoRunSummary.progress` should follow this shape:

```json
{
  "progress": {
    "schema_version": "public_demo_progress_v1",
    "current_stage": "ready_for_confirmation",
    "current_label": "推荐方案已准备好",
    "stage_history": [
      "understanding_request",
      "planning_queries",
      "searching_activities",
      "searching_dining",
      "checking_availability",
      "building_itinerary",
      "checking_route_time",
      "reviewing_plan",
      "ready_for_confirmation"
    ]
  }
}
```

Stable label mapping for `current_label`:

```json
{
  "understanding_request": "正在理解需求",
  "planning_queries": "正在规划查询",
  "searching_activities": "正在查询游玩地点",
  "searching_dining": "正在查询餐厅",
  "checking_availability": "正在检查营业与可用性",
  "building_itinerary": "正在组合行程",
  "checking_route_time": "正在计算路线与时间",
  "reviewing_plan": "正在复核方案",
  "ready_for_confirmation": "推荐方案已准备好",
  "executing_confirmed_actions": "已确认，正在执行动作"
}
```

Notes:

- `stage_history` is a public-safe summary of reached milestones, not a raw workflow trace.
- top-level `status`, `clarification`, `plan_version`, and plan payloads remain unchanged in this task.
- the contract is additive only.

## 6. Observability

This task consumes existing workflow evidence and produces one public-safe derived summary. It does not add a new telemetry backend.

It may read:

- persisted demo metadata
- persisted run status
- persisted tool-event rows

It must not expose through `progress`:

- `node_history`
- raw workflow node names
- raw tool names
- `tool_event_id`
- `trace_id`
- `session_id`
- provider-specific payloads
- prompts
- auth headers
- secrets
- raw debug fields

This task also must not require Redis progress-stream keys for successful readback. A run that already completed must still be able to reconstruct its public progress snapshot from durable data only.

## 7. Failure Handling

- If `initial_node_history` is missing or malformed, use the safest status-based fallback stage instead of failing the demo response.
- If `continuation_history` is missing or malformed, do not assume execution; fall back to non-execution stage resolution.
- If search tool events are missing, malformed, or ambiguous, do not invent both search stages. Keep the last safely supported stage instead.
- If an unknown workflow node name appears in stored history, ignore it rather than surfacing it publicly.
- If progress reconstruction fails unexpectedly, the request must still return the existing demo summary and use a coarse fallback stage derived from `run.status` and execution state.
- Clarification runs must continue returning the existing `clarification` object even when progress reconstruction falls back.
- Declined runs must never be misreported as `executing_confirmed_actions`.
- AMAP preview runs must continue to stop at the confirmation boundary and must not be misreported as executing.

## 11. Notes for the Implementer

Keep this task contract-first and backend-only.

The key implementation decision is to compute `DemoRunSummary.progress` at read time from existing persisted evidence instead of widening the task into new runtime transport. That keeps the scope small, keeps the task independently reviewable, and gives the later async/polling task a stable contract to reuse.

If the implementation starts to require:

- asynchronous `POST /demo/runs`
- new background execution infrastructure
- a new polling endpoint
- SSE or WebSocket streaming
- frontend chat rendering changes

stop and split that into a follow-up task instead of widening `078`.
