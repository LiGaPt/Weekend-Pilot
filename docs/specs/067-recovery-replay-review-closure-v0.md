# Spec: 067 Recovery Replay Review Closure v0

## 1. Goal

Close the remaining reviewer-verification gap around WeekendPilot's existing recovery benchmark stack by turning the already-implemented failure injection, replay harness, and internal recovery review surfaces into one direct, deterministic closure workflow.

The repository already has all three ingredients separately:
- benchmark-mode failure injection
- file-based replay of sanitized benchmark case reports
- internal recovery-path review built from persisted run metadata

What is still missing is one canonical reviewer path that proves those pieces connect correctly for a real recovery case without requiring manual Python snippets, manual path chasing, or ad hoc run-to-run interpretation. After this task, a reviewer must be able to run one command and get one compact evidence bundle for a single standard failure case that proves:
- the failure path executed as expected
- the replay harness successfully replayed the written benchmark case report
- the internal recovery review metadata points back to that same source report

## 2. Project Context

`docs/PROJECT_BLUEPRINT.md` explicitly calls out both **Failure injection and replay harness** and broader harness engineering responsibilities. It also requires recovery behavior to stay bounded, traceable, benchmark-visible, and confirmation-safe.

`docs/NEXT_PHASE_ROADMAP.md` places remaining recovery work under milestone `M5. 恢复、真实 provider、记忆治理`, while the repository is already ahead on the infrastructure that would normally precede this task:

- Task `029` made benchmark failure injection real for `family_route_failure_v1`.
- Task `030` added the file-based replay harness for sanitized benchmark case reports.
- Task `042` surfaced `recovery_path_summary` on internal observability and linked recovery runs to benchmark report paths.
- Task `051` upgraded bounded recovery behavior to v1 while keeping it typed and observable.
- Task `052` added composite chaos cases and replay-stable failure-chain signatures.
- Task `061` added one-click formal verification for `all_registered`.
- Task `065` added the blocking `release_gate_v1`.
- Task `066` closed the memory-governance documentation gap and confirmed that the broader benchmark/review infrastructure is already in place.

That means the next smallest real gap is not more benchmark infrastructure or more recovery logic. The gap is reviewer closure: the repository still lacks one canonical command and one canonical artifact that connect a real failure case, a replay run, and the internal recovery review evidence into one directly checkable chain.

This task therefore stays narrow and closes the minimal slice of roadmap item `恢复路径可视化与 replay 联动` by using one existing standard failure case:
- `family_route_failure_v1`

## 3. Requirements

### A. Add one canonical recovery-review closure runner

- Add a new module:
  - `backend/app/benchmark/recovery_review.py`
- The module must provide a typed runner function with the same local-review ergonomics as `release_gate.py` and `formal_verification.py`.
- The runner must use existing repository capabilities only:
  - `BenchmarkHarness`
  - `BenchmarkReplayHarness`
  - `InternalObservabilityService`
- The default and only required v0 review case is:
  - `family_route_failure_v1`
- The runner must not require a running FastAPI server.
- The runner must read internal recovery review data directly from persisted database metadata through `InternalObservabilityService`.

### B. Produce one deterministic evidence bundle

- The runner must create a unique output directory under:
  - `var/recovery-reviews/`
- The directory name must follow a stable review-run prefix such as:
  - `recovery-review-<uuid>`
- The runner must write:
  - the source benchmark case report generated by `BenchmarkHarness.run_case(...)`
  - the replay report generated by `BenchmarkReplayHarness.replay_report(...)`
  - one aggregate recovery-review artifact JSON
  - one local trace buffer file for the source benchmark run
- The aggregate recovery-review artifact must use a stable schema version:
  - `weekendpilot_recovery_replay_review_v1`
- The runner must also refresh one stable latest alias on success only:
  - `var/recovery-reviews/latest-family_route_failure_v1-review.json`
- Failed or errored review runs must keep their unique run directory for inspection.
- Failed or errored review runs must not overwrite the latest alias.

### C. Lock the canonical source failure path

- The source run must execute the existing benchmark case:
  - `family_route_failure_v1`
- The runner must drive the source case through `BenchmarkHarness.run_case(...)`, not through a fake or shortcut path.
- The source result must be treated as the canonical failure-path evidence only when all of these stay true:
  - benchmark `status == "passed"`
  - workflow `workflow_status == "failed"`
  - `action_count == 0`
  - `failure_chain_summary.profile_id == "route_unavailable_v0"`
  - `failure_chain_summary.injected_effects == ["check_route:route_infeasible:failed"]`
  - `failure_chain_summary.recovery_actions == ["stop_safely"]`
  - `failure_chain_summary.bounded == true`
- The runner must treat any drift from that contract as a failed or errored review outcome.
- This task must not add new benchmark cases, new failure profiles, or new benchmark suite memberships.

### D. Lock the canonical replay path

- Replay must consume the written source benchmark case report from disk.
- The runner must call:
  - `BenchmarkReplayHarness.replay_report(source_report_path)`
- The runner must not use the in-memory replay shortcut as the canonical reviewer path.
- The replay result must be treated as valid closure evidence only when all of these stay true:
  - replay `status == "passed"`
  - replay `mismatches == []`
  - replay `replay_benchmark_status == "passed"`
  - replay `source.workflow_status == "failed"`
  - replay `replay.workflow_status == "failed"`
  - replay `source.failure_chain_signature == ["check_route:route_infeasible:failed"]`
  - replay `replay.failure_chain_signature == ["check_route:route_infeasible:failed"]`
- Replay output must remain sanitized and file-based under the recovery-review output tree.

### E. Lock the internal recovery review evidence chain

- The runner must load the internal observability summary for the source run via:
  - `InternalObservabilityService.get_run_summary(source_run_id)`
- The aggregate review artifact must include a compact recovery-review summary derived from that internal observability response.
- The recovery-review summary must only be treated as valid closure evidence when all of these stay true:
  - `benchmark_artifact_summary` is present
  - `benchmark_artifact_summary.case_id == "family_route_failure_v1"`
  - `benchmark_artifact_summary.report_path == source_report_path`
  - `recovery_path_summary` is present
  - `recovery_path_summary.attempt_count == 1`
  - `recovery_path_summary.max_attempts == 2`
  - the first recovery attempt has `recovery_action == "stop_safely"`
  - the first recovery attempt has `status == "stopped"`
  - `recovery_path_summary.replay_source.case_id == "family_route_failure_v1"`
  - `recovery_path_summary.replay_source.benchmark_report_path == source_report_path`
- This exact path match is the critical closure invariant for this task:
  - source benchmark report path
  - internal benchmark artifact report path
  - internal recovery replay source path
  - replay harness input path
  must all refer to the same source report.

### F. Add one aggregate review artifact with explicit checks

- The aggregate review artifact must include exactly these named checks:
  - `benchmark_failure_path`
  - `replay_matches_source_report`
  - `observability_links_source_report`
- Each check must record:
  - its stable name
  - `passed: true|false`
  - one short reviewer-readable detail string
- The aggregate review artifact status must be:
  - `passed` when all three checks pass
  - `failed` when the run completes but one or more checks fail
  - `error` when benchmark execution, replay execution, observability lookup, or artifact writing cannot complete
- The aggregate review artifact must include:
  - `case_id`
  - `run_id`
  - `run_directory`
  - `source_report_path`
  - `replay_report_path`
  - `latest_review_path`
  - `failure_chain_summary`
  - compact replay summary
  - compact recovery review summary
  - the three explicit checks

### G. Add one reviewer-facing CLI entry point and docs

- Add a new script:
  - `scripts/run_recovery_replay_review.py`
- The script must mirror the style of the existing release-gate and formal-verification scripts:
  - ensure repo root is on `sys.path`
  - call `backend.app.benchmark.recovery_review.main`
- The module `main()` implementation must:
  - start `postgres` and `redis` through `docker compose up -d postgres redis`
  - wait for both services
  - run `python -m alembic upgrade head`
  - execute the recovery-review runner
  - print a concise summary with source path, replay path, review artifact path, and run ID
  - exit `0` only when the review artifact status is `passed`
  - exit `1` for `failed` or `error`
- Update `README.md` with one concise `Recovery Replay Review` section.
- Update `docs/WEB_DEMO_README.md` so it states that the new script produces a run ID and a source benchmark report path that can be cross-checked against the existing internal observability surface.
- Documentation must state clearly that this task closes one canonical reviewer path only and does not add replay browsing UI.

### H. Keep the task additive and narrow

- Do not change `family_route_failure_v1`.
- Do not change `route_unavailable_v0`.
- Do not change replay comparison rules in `backend/app/benchmark/replay.py`.
- Do not change internal observability route shapes or frontend pages.
- Do not change benchmark suite IDs, suite counts, release-gate rules, or formal-verification rules.
- Do not add new dependencies.
- Do not add migrations.
- Do not commit generated `var/` artifacts.

## 6. Observability

This task must not add a new telemetry backend, new persisted database fields, or new API routes.

It must reuse existing persisted and sanitized evidence surfaces:

- benchmark case report JSON written by `BenchmarkHarness`
- replay case report JSON written by `BenchmarkReplayHarness`
- `agent_runs.metadata_json["benchmark"]["artifact_summary"]`
- `agent_runs.metadata_json["workflow"]["recovery"]`
- `InternalObservabilityService.get_run_summary(...)`

The new aggregate review artifact is itself a local file-based evidence product under `var/recovery-reviews/`. It must remain sanitized and must not include:

- `action_id`
- `tool_event_id`
- raw request or response payload dumps beyond existing safe summaries
- API keys
- tokens
- secrets
- authorization headers
- prompts
- tracebacks
- stack traces

## 7. Failure Handling

- If `family_route_failure_v1` cannot be loaded, the runner must return `status="error"`.
- If source benchmark execution returns `status="error"` or lacks `run_id` or `report_path`, the runner must return `status="error"`.
- If source benchmark execution completes but violates the canonical failure-path contract, the aggregate review must return `status="failed"`.
- If replay execution returns `status="error"`, the aggregate review must return `status="error"`.
- If replay execution completes but has mismatches, the aggregate review must return `status="failed"`.
- If internal observability lookup fails, the aggregate review must return `status="error"`.
- If internal observability summary exists but lacks `benchmark_artifact_summary` or `recovery_path_summary`, the aggregate review must return `status="failed"`.
- If internal observability report paths do not match the source report path exactly, the aggregate review must return `status="failed"`.
- If artifact writing fails, the runner must raise or return a typed review error.
- If the review status is `failed` or `error`, the latest alias must not be overwritten.
- Existing PostgreSQL, Redis, benchmark, replay, and internal-observability error behavior outside this new closure layer must remain unchanged.

## 11. Notes for the Implementer

Keep this task tightly scoped to one reviewer-visible closure flow.

The repository already has the necessary primitives. The point of this task is not to build new recovery behavior, new replay logic, or new benchmark cases. The point is to prove that the existing primitives line up on one canonical case and to make that proof easy for a reviewer to run again.

Important defaults chosen here:

- the only required v0 case is `family_route_failure_v1`
- the canonical replay path must replay the written source report from disk
- the canonical recovery review path must use persisted metadata through `InternalObservabilityService`
- the closure invariant is path equality across source report, internal benchmark artifact summary, internal recovery replay source, and replay input
- failed review runs keep their unique run directories but do not refresh the latest alias

If implementation pressure starts pulling in arbitrary case selection, UI work, replay browsing, or refactoring the other benchmark runners, stop and narrow the task back down.
